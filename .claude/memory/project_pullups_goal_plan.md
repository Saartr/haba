## Цель

Новая категория solo-цели `pullups` — в отличие от шагов (числовой порог в день) и курения (boolean чек-ин), это **прогрессивная программа**: план растёт от текущей формы к конечной цели по неделям, с переходом на следующую тренировку при выполнении и полным пересчётом плана при невыполнении.

**Why:** Пользователь хочет цель, которая адаптируется под реальный прогресс, а не статичный ежедневный план. Решение принято в сессии 2026-06-20 после уточняющих вопросов и разбора макетов Figma (frame `1424:3596`, файл `yTlBE7MQqQTvxez3UP4n48`).

**Scope:** только персональная (solo) цель в этой итерации. Группового режима для подтягиваний нет — [[project_main_screen]] и архитектура групп не трогаются.

**How to apply:** При реализации — заводить новую категорию `pullups` параллельно с `smoking`/`steps`, не трогая их логику. Числовые поля (форма/цель) брать в диапазоне 1–30.

---

## Данные: новые колонки в `habits`

Решение: расширить существующую таблицу `habits` нуллабельными колонками (не отдельная таблица) + сохранённый план целиком, без пересчёта при каждом рендере экрана.

```sql
ALTER TABLE habits ADD COLUMN IF NOT EXISTS current_form INT NULL;       -- старт: повторений в подходе
ALTER TABLE habits ADD COLUMN IF NOT EXISTS target_reps INT NULL;        -- финал: повторений в подходе (при 4 подходах)
ALTER TABLE habits ADD COLUMN IF NOT EXISTS intensity TEXT NULL;         -- 'low' | 'medium' | 'high'
ALTER TABLE habits ADD COLUMN IF NOT EXISTS training_days SMALLINT[] NULL; -- 1=Пн..7=Вс, длина = N тренировок/неделю
ALTER TABLE habits ADD COLUMN IF NOT EXISTS pullups_plan JSONB NULL;     -- [{session, sets, reps}], индекс = порядковая тренировка
ALTER TABLE habits ADD COLUMN IF NOT EXISTS pullups_session_index INT NOT NULL DEFAULT 0; -- указатель на текущую тренировку в plan
```

`pullups_plan` индексируется **по тренировкам, а не по календарным неделям** — так пересчёт после невыполнения не зависит от того, сколько календарных дней прошло, только от `pullups_session_index`.

`category = 'pullups'` в `CATEGORY_OPTIONS` в `app/(tabs)/create-habit.tsx` (сейчас там только `{label: 'Курение', value: 'smoking'}`).

**How to apply:** Миграция — новый файл в `backend/src/db/` (по аналогии с `migrate_habits.js`), `IF NOT EXISTS` на каждой колонке, коммит в `backend/`, деплой через `./deploy-backend.ps1` ([[feedback_ssh_file_edit]]).

## Таблица интенсивности

| Интенсивность | Тренировок/неделю | Темп роста объёма/неделю |
|---|---|---|
| Низкая | 2 | +7% |
| Средняя | 3 | +10% |
| Высокая | 4 | +13% |

`training_days` должен содержать ровно столько дней, сколько тренировок/неделю у выбранной интенсивности — валидируется на форме создания (см. Multiselect ниже).

## Формула расчёта плана (тренерская валидация пройдена)

**Шаг 1 — число недель:**
```
объём_старт  = 3 подхода × current_form
объём_финал  = 4 подхода × target_reps
n_недель = ceil( log(объём_финал / объём_старт) / log(1 + темп_роста) )
```

**Шаг 2 — объём тренировки на каждой неделе k (0..n_недель):**
```
объём_k = round(объём_старт × (1 + темп_роста)^k), capped на объём_финал на последней неделе
```
Все тренировки одной недели имеют одинаковый целевой объём (план не дробится по дням внутри недели — проще и предсказуемее для пользователя).

**Шаг 3 — раскладка объёма на подходы × повторения:**
```
если объём_k / 3 <= 8:           sets = 3, reps = ceil(объём_k / 3)
иначе если объём_k / 4 <= target_reps: sets = 4, reps = ceil(объём_k / 4)
иначе (последняя неделя):        sets = 4, reps = target_reps
```

**Почему это нормально с точки зрения тренера (валидация, которую запросил пользователь):**
Это стандартная схема прогрессии в калистенике: сначала растут **повторения в подходе** (безопаснее на старте, не требует доп. объёма за тренировку), и только когда подход становится «тяжёлым» (~8 повторений — типичный порог рабочего подхода) добавляется **4-й подход**, а не дальнейшее наращивание повторений в одном сете (так избегаем срыва техники/перегрузки одного подхода). После выхода на 4 подхода рост идёт через повторения до финальной цели — к этому моменту накоплена силовая выносливость, это безопасно. Возможное временное снижение reps/подход в момент перехода 3→4 подхода (объём растёт, но делится на больше подходов) — это нормальная практика периодизации объёма, не баг.

`pullups_plan` хранит развёрнутый по тренировкам массив (длина = n_недель × тренировок/неделю), а не по неделям — каждая запись `{session: i, sets, reps}` использует объём_k недели, в которую попадает тренировка `i`.

## Прогресс и пересчёт

- `pullups_session_index` — указатель на следующую тренировку в `pullups_plan`.
- Кнопки **бинарные** — «Выполнил» / «Не выполнил», без числового ввода фактического результата (подтверждено пользователем 2026-06-20, снимает более раннюю неоднозначность плана).
- **Выполнил** → `pullups_session_index += 1`, переход к следующей тренировке плана.
- **Не выполнил** → полный пересчёт плана со сдвигом вперёд:
  - новый `current_form` = объём **последней успешно выполненной тренировки** (`pullups_plan[pullups_session_index - 1]`, переведённый обратно в «повторений в подходе» при 3 подходах: `round(объём / 3)`); если провал на самой первой тренировке — `current_form` не меняется (база — исходное значение при создании цели)
  - план пересчитывается от этого `current_form` к тому же `target_reps` той же интенсивностью (формула выше, `n_недель` и раскладка заново) — план становится длиннее, т.е. «сдвигается» вперёд по времени
  - `pullups_session_index` сбрасывается на 0 нового плана
- Уведомление о пересчёте — **снэкбар/баннер в приложении** (`lib/snackbar-context`), не push (push не реализован — [[project_pending_features]]).
- День без тренировки (не входит в `training_days`) — `inactive` в `Calendar.tsx` (уже поддерживается, рендерит без иконки).

## UI: форма создания (`app/(tabs)/create-habit.tsx`)

Ветка `category === 'pullups'` (параллельно существующей `smoking`-ветке, та не трогается):
- **Текущая форма** — числовой пикер 1–30 (повторений в подходе сейчас)

- **Интенсивность** — `Select` (Низкая/Средняя/Высокая)
- **Дни недели** — новый компонент `Multiselect` (см. ниже), количество выбираемых дней жёстко = тренировок/неделю выбранной интенсивности

`CATEGORY_OPTIONS` дополняется `{ label: 'Подтягивания', value: 'pullups' }`.

## Новый компонент: `components/Multiselect.tsx`

Решение пользователя: по аналогии с `components/Select.tsx`, новый компонент, **без макета в Figma** (явное разрешение на это единственное исключение из правила [[feedback_figma_source_of_truth]]).

Отличия от `Select.tsx`:
- `value: string[]` вместо одиночного `value`
- чекбокс-индикатор у каждой строки вместо одной галочки
- кнопка **«Готово»** снизу `BottomSheet` — закрывает шит, коммитит выбор (а не закрытие по тапу на строку, как в `Select`)
- проп `exactCount: number` — жёсткое ограничение: когда выбрано `exactCount` элементов, остальные строки визуально дизейблятся (тап неактивен), пока пользователь не снимет одну из выбранных
- при смене интенсивности на форме создания (другое `exactCount`) — текущий выбор дней нужно сбросить, если он не совпадает по длине

**How to apply:** Строить по структуре `Select.tsx` (`BottomSheet` + `FlatList`, `backgroundColor: item.value === ... ? c.surface.cardGrey : 'transparent'`), добавляя множественность и лимит выше.

## UI: экран цели (`app/(tabs)/habit/[id].tsx`, ветка `category === 'pullups'`)

Три состояния по макетам Figma:
1. **Тренировочный день** — текущая запланированная тренировка (`pullups_plan[pullups_session_index]`): «N подходов по M повторений», кнопки Выполнил/Не выполнил
2. **День отдыха** — день не входит в `training_days`, без карточки тренировки (Calendar рендерит `inactive`, без иконки)
3. **Цель достигнута** — `pullups_session_index >= pullups_plan.length`. Не хватает CTA «Поставить новую цель» в макете — реализовать как `Button`, переход на `create-habit` (разумный дефолт, не блокирующий вопрос)

«⋮»-меню — по аналогии с существующим solo-меню (`Редактировать` / `Удалить`, см. `app/(tabs)/habit/[id].tsx`).

Календарь — переиспользуется `components/Calendar.tsx`, но статус дня для pullups считается иначе, чем для шагов (`buildDays()` сейчас сравнивает `loggedValue >= goalValue` — числовой порог). Для pullups нужна параллельная ветка: `check` = тренировка дня выполнена, `miss` = тренировочный день пропущен, `inactive` = день отдыха, `current`/`future` — как есть.
